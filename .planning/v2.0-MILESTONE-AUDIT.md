## Remaining Gaps (3)

### 1. ANLZ-02: Catalog selection has no effect on analysis

**Severity:** Medium — UI collects data that is silently ignored
**Location:** `frontend/src/app/(app)/projekte/[id]/analyse/client.tsx` line 213

The wizard step 2 lets users select a catalog. `state.catalogId` is stored and validated. However, `handleStartAnalysis` sends only `{ project_id: prepareResult.pythonProjectId }` to Python. The `catalogId` is never included. Python's `AnalyzeProjectRequest` has no `catalog_id` field — it uses whatever catalog is currently activated in memory.

**Impact:** If multiple catalogs exist, the user's selection is meaningless. Currently mitigated by most deployments having only one catalog.

### 2. DASH-04: /neue-analyse page renders stub

**Severity:** Low-Medium — UX dead end with easy workaround
**Location:** `frontend/src/app/(app)/neue-analyse/page.tsx`

The dashboard "Neue Analyse starten" button, sidebar link, and keyboard shortcut `N` all navigate to `/neue-analyse`. This page still renders a Phase 10 placeholder ("Analyse-Wizard wird in Phase 13 implementiert"). The actual wizard is only accessible via `/projekte/[id]/analyse`.

**Impact:** Users clicking quick-action reach a dead end. They must navigate to a project first.

### 3. AUTH-05: Wrong auth redirect path in 3 pages

**Severity:** Low — only triggers on edge cases (session expiry during page load)
**Location:** Three files redirect to `/auth/login` instead of `/login`:
- `frontend/src/app/(app)/projekte/page.tsx` line 16
- `frontend/src/app/(app)/projekte/[id]/page.tsx` line 15
- `frontend/src/app/(app)/projekte/[id]/analyse/page.tsx` line 17

**Impact:** 404 instead of login page when session check fails in these server components. The `(app)/layout.tsx` guard catches most cases before these pages render.
