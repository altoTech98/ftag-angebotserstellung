**Part A: Extend backend/routers/catalog.py with new endpoints:**

Keep existing `/catalog/info` and `/catalog/upload` endpoints. Add:

1. **GET `/catalog/products`** -- Search and filter products with pagination.
   - Query params: `search` (text), `category` (filter), `page` (int, default 1), `limit` (int, default 50)
   - Uses `get_catalog_index()` to search `all_profiles` by `compact_text` and `category`
   - Returns `{ products: [...], total, page, limit, pages }` where each product has `row_index`, `category`, `summary` (compact_text), `fields` (key_fields), `kostentraeger` (from key_fields or extracted from df)
   - See research code example for implementation pattern

2. **POST `/catalog/validate`** -- Validate a catalog file from blob URL.
   - Body: `{ blob_url: string }` or accepts raw file bytes
   - Downloads from blob URL, parses with `pd.read_excel(header=6)`
   - Validates: has expected columns (at least Produktegruppen at col 0), min 10 rows, no fully empty rows
   - Returns `{ valid: bool, total_rows: int, main_products: int, categories: int, errors: [], warnings: [] }`
   - Does NOT activate -- just validates

3. **POST `/catalog/activate`** -- Activate a catalog version.
   - Body: `{ blob_url: string }` (URL to the Excel file in Vercel Blob)
   - Downloads Excel from blob URL, writes to `data/produktuebersicht.xlsx` (with backup of existing), calls `invalidate_catalog_cache()` + `get_catalog_index()`
   - Returns catalog info (same as `/catalog/info`)
   - On failure, restore backup (like existing upload endpoint logic)

4. **POST `/catalog/diff`** -- Compare two catalog versions.
   - Body: `{ old_blob_url: string, new_blob_url: string }`
   - Downloads both, parses with pandas, compares row counts and categories
   - Returns `{ old_count, new_count, added, removed, category_changes: { added: [], removed: [] } }`

5. **GET `/catalog/products/{row_index}`** -- Get full product details.
   - Uses `get_catalog_index().get_product_extended(row_index)`
   - Returns the extended product dict

6. **POST `/catalog/products/override`** -- Apply a product override (for KAT-04).
   - Body: `{ product_key: string, action: "edit"|"add"|"delete", data: {...} }`
   - For "edit"/"add": merges override data into the catalog index at query time (or stores for later merge)
   - For now, just returns success -- actual override application happens on Next.js side via Prisma. Python does not need to persist overrides; it only needs to know to include them when serving products.
   - This endpoint is optional for MVP; overrides can be handled purely in Next.js server actions.

For downloading blob URLs, use `httpx` (already available) or `urllib.request` to fetch the file content. Add `import io` for BytesIO parsing.

**Part B: Create frontend/src/lib/actions/catalog-actions.ts:**

Follow the exact same pattern as `project-actions.ts`:
- `'use server'` directive
- Import prisma, auth, headers, revalidatePath
- Use `PYTHON_BACKEND_URL` and `PYTHON_SERVICE_KEY` from `process.env`

Server actions to create:

1. **`getCatalogs()`** -- List all catalogs with their active version info.
   - Permission: `catalog.read`
   - Query: `prisma.catalog.findMany({ include: { versions: { where: { isActive: true } } } })`
   - Returns catalog list

2. **`uploadCatalog(formData: FormData)`** -- Upload new catalog file.
   - Permission: `catalog.upload`
   - Steps: (a) Upload file to Vercel Blob (`put('catalogs/...')`), (b) POST to Python `/api/catalog/validate` with blob_url, (c) if valid, find or create Catalog record in Prisma, (d) create CatalogVersion record with version number, blob URL, validation stats, (e) if first version or user confirms, activate it (set isActive, call Python `/api/catalog/activate`), (f) revalidatePath('/katalog')
   - Returns `{ success, validation, catalogId, versionId }` or `{ error, validation }`

3. **`getCatalogProducts(catalogId: string, search?: string, category?: string, page?: number)`** -- Browse products.
   - Permission: `catalog.read`
   - Fetches from Python via BFF: `GET /api/catalog/products?search=...&category=...&page=...&limit=50`
   - Also loads ProductOverride records from Prisma for this catalog to merge into results
   - Returns products with override status marked

4. **`getCatalogVersions(catalogId: string)`** -- Get version history.
   - Permission: `catalog.read`
   - Query: `prisma.catalogVersion.findMany({ where: { catalogId }, orderBy: { versionNum: 'desc' } })`

5. **`rollbackVersion(catalogId: string, versionId: string)`** -- Rollback to a previous version.
   - Permission: `catalog.update`
   - Steps: (a) Set current active version `isActive: false`, (b) set target version `isActive: true`, (c) update catalog `activeVersionId`, (d) POST to Python `/api/catalog/activate` with target version's blobUrl, (e) revalidatePath('/katalog')

6. **`compareVersions(oldVersionId: string, newVersionId: string)`** -- Get diff between two versions.
   - Permission: `catalog.read`
   - Load both versions from Prisma (get blob URLs), POST to Python `/api/catalog/diff`
   - Returns diff results

7. **`saveProductOverride(catalogId: string, productKey: string, action: string, data?: Record<string, unknown>)`** -- Save a product edit/add/delete.
   - Permission: `catalog.update`
   - Upsert `prisma.productOverride.upsert({ where: { catalogId_productKey: { catalogId, productKey } }, ... })`
   - revalidatePath('/katalog')

8. **`deleteProductOverride(overrideId: string)`** -- Remove a product override (revert to original).
   - Permission: `catalog.update`
   - `prisma.productOverride.delete({ where: { id: overrideId } })`
   - revalidatePath('/katalog')
